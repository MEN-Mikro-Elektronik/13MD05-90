Using OSS Signals

Signals are used to notify processes/tasks in user land about asynchronous event inside (MDIS) drivers. This releases an application from periodically polling for that event.

Before a signal can be sent by the driver to the application, the signal condition must have been activated by the application. This is normally done using a M_setstat call. Each driver can define any number of signal conditions appropriate for that device.

The driver's interrupt or alarm routine will check the installed conditions and -- if the condition is satisfied -- send the signal to the process that installed it. When the application process receives the signal, it invokes a signal handler routine, asynchronous to the processes current execution thread. Therefore, from the application process point of view, a signal handler can be compared with an interrupt routine, because it interrupts the program's normal execution flow.

Not all OSes that run MDIS natively support signals, only the UNIX like operating systems know the signal concept. On the other systems (RTAI, Windows and Ardence RTX), signals are emulated with some other interprocess communication method and a dedicated thread that invokes the signal handler.

To allow application processes to be written OS independently, MDIS provides common API routines in the USR_OSS library that hides the implementation details from the application.

Typical usage:

      // Driver (any call):
          OSS_SigCreate();
      // Driver IRQ:
          OSS_SigSend();
      // Driver (any call):
          OSS_SigRemove();
